# Generated BLOB/TEXT Prefix Indexes

## Problem

MyLite supports ordinary BLOB/TEXT row payloads, bounded BLOB/TEXT prefix
indexes, basic generated columns, and generated-column indexes over scalar
values. The remaining combined gap is generated columns whose values are
BLOB/TEXT and whose generated key parts use bounded prefix indexes.

This matters because MariaDB and MySQL schemas commonly normalize or derive
searchable text into generated columns before indexing a prefix. MyLite should
support the subset where MariaDB computes the generated value and the resulting
prefix key image fits the current MyLite index-entry page.

## Source Findings

- MariaDB base: `mariadb-11.8.6` /
  `9bfea48ce1214cc4470f6f6f8a4e30352cef84e7`.
- `mariadb/sql/table.cc`: generated-column expressions are unpacked from the
  table-definition image, and `TABLE::update_keypart_vcol_info()` reconnects
  generated-column metadata to key parts after reopen.
- `mariadb/sql/table.cc`: `TABLE::update_virtual_fields()` computes generated
  values before write and after reads when virtual values are needed.
- `mariadb/sql/key.cc`: `key_copy()` builds index key tuples from MariaDB row
  buffers, and `key_tuple_cmp()` / `key_buf_cmp()` compare those key images.
- `mariadb/sql/field.cc`: BLOB/TEXT prefix key images are generated by the
  field implementation, including bounded length and collation-aware text
  handling.
- `mariadb/storage/mylite/ha_mylite.cc`: MyLite now accepts generated key
  parts and bounded BLOB/TEXT prefix key parts independently, stores opaque
  MariaDB key bytes in index-entry pages, and rejects unsupported key classes.

## Scope

- MyLite-routed `ENGINE=InnoDB` tables with virtual and stored generated
  BLOB/TEXT columns.
- Bounded secondary and unique prefix indexes on generated BLOB/TEXT columns
  declared in initial `CREATE TABLE`.
- Insert, duplicate-prefix rejection, forced-index lookup, update/delete
  visibility, close/reopen visibility, catalog metadata, and sidecar gates.

## Non-Goals

- Full-length unbounded generated BLOB/TEXT indexes.
- Oversized generated key images that exceed the current MyLite index-entry
  page limit.
- Expression/hidden generated indexes, FULLTEXT, SPATIAL, hash, vector, or
  foreign-key indexes. Generated primary keys follow MariaDB's SQL-layer
  rejection policy.
- Online or default-algorithm generated BLOB/TEXT prefix index DDL.
- SQL rollback, savepoints, crash recovery beyond current statement
  checkpointing, or broad generated expression matrices.

## Design

No new storage format is introduced. The supported path composes existing
behavior:

1. MariaDB computes generated BLOB/TEXT values before handler writes.
2. MyLite serializes generated stored values in normal row payloads and lets
   MariaDB recompute virtual values from restored base row buffers.
3. MyLite calls `key_copy()` for generated prefix indexes, receiving MariaDB's
   bounded BLOB/TEXT key image.
4. Existing MyLite index-entry pages store those opaque key bytes.
5. Existing row-state filtering hides stale generated prefix index entries
   after update/delete.

## Compatibility Impact

Generated-column support expands to generated BLOB/TEXT values and bounded
generated BLOB/TEXT prefix indexes declared in initial DDL. A follow-up slice
extends the same key class to standalone copy-rebuild index DDL. The area
remains partial because full or oversized generated BLOB/TEXT indexes,
expression/hidden indexes, SQL rollback, and broad expression matrices remain
planned; generated primary keys follow MariaDB's SQL-layer rejection policy.

## DDL Metadata Routing Impact

Initial `CREATE TABLE` can publish catalog metadata for generated BLOB/TEXT
prefix indexes when the key shape passes the existing MyLite support predicate.
Unsupported generated key classes must still fail before catalog publication.

## Single-File And Embedded-Lifecycle Impact

Generated BLOB/TEXT row payloads and generated prefix index entries live in the
primary `.mylite` file. No durable MariaDB sidecar, plugin-owned table file, or
runtime schema directory should be required after close/reopen.

## Public API, File-Format, Size, And Dependency Impact

No public API, file-format, dependency, or embedded profile change is expected.
The slice should add compatibility coverage and documentation only unless the
existing support predicates or key-copy path need correction.

## Test And Verification Plan

- Extend storage-engine smoke coverage with a routed table that has:
  - a stored generated `TEXT` column with a secondary prefix index,
  - a virtual generated `TEXT` column with a unique prefix index,
  - a stored generated `BLOB` column with a secondary prefix index.
- Verify forced-index reads over generated text and binary prefix indexes.
- Verify duplicate-prefix rejection through the unique generated prefix index.
- Verify update/delete filtering and reuse after the old generated prefix
  becomes non-live.
- Verify catalog metadata and generated prefix reads after close/reopen.
- Run generated-column, unsupported-index, routed DDL/DML, sidecar, format,
  tidy, preset, and diff checks.

## Acceptance Criteria

- Bounded generated BLOB/TEXT prefix indexes declared in initial DDL are
  accepted for supported routed table shapes.
- Forced-index reads find rows through generated BLOB/TEXT prefix entries
  before and after close/reopen.
- Unique generated prefix indexes reject duplicate live prefixes and allow reuse
  after update/delete makes the old prefix non-live.
- Compatibility docs narrow the remaining BLOB/TEXT generated-column work to
  full/oversized indexes and broader expression matrices.

## Risks And Open Questions

- Standalone generated BLOB/TEXT prefix index DDL is covered by the follow-up
  generated BLOB/TEXT prefix index DDL slice.
- BLOB/TEXT generated expressions beyond deterministic string/binary functions
  need broader MariaDB comparison coverage.
