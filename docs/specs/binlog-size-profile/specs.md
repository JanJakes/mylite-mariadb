# Binlog Size Profile

## Problem Statement

The MyLite minsize profile still links MariaDB's binary-log transaction
participant and event machinery even though MyLite's default embedded product
does not support replication, relay logs, file-based binlogs, or replaying
binary-log events into the local database.

The visible built-in plugin named `binlog` is not the main size issue. It is a
daemon plugin wrapper around a `transaction_participant` used by MariaDB's
commit, rollback, savepoint, XA, and row-logging paths. This slice researches
that boundary and defines the safe near-term size work: keep the transaction
participant intact for now, reject user-facing binlog/replay surfaces
explicitly where possible, and prefer link-time dead-code elimination attempts
over broad source deletion until the transaction manager and log-event
interfaces are separated deliberately.

## Source Findings

MariaDB source references are from the imported MariaDB Server tag
`mariadb-11.8.6` (`9bfea48ce1214cc4470f6f6f8a4e30352cef84e7`).

- `vendor/mariadb/server/sql/sql_builtin.cc.in` hard-codes
  `builtin_maria_binlog_plugin` into `mysql_mandatory_plugins` before the
  generated plugin list.
- `vendor/mariadb/server/sql/log.cc` defines the global
  `MYSQL_BIN_LOG mysql_bin_log(&sync_binlog_period)` and the global
  `transaction_participant binlog_tp`.
- `vendor/mariadb/server/sql/log.cc` declares `maria_declare_plugin(binlog)`.
  `binlog_init()` initializes `binlog_tp`, attaches it to the plugin data, and
  calls `setup_transaction_participant(plugin)`.
- `vendor/mariadb/server/sql/handler.cc` has direct transaction-manager checks
  for `&binlog_tp`. Commit and rollback order binlog first through
  `run_binlog_first()`, compare handler participants against `&binlog_tp`, and
  call `binlog_commit_by_xid()` / `binlog_rollback_by_xid()` in XA paths.
- `vendor/mariadb/server/sql/log.cc` implements
  `THD::binlog_setup_trx_data()`, `THD::binlog_start_trans_and_stmt()`, cache
  savepoint callbacks, GTID/binlog position helpers, and row-event write paths
  that reference `Gtid_log_event`, `Query_log_event`, `Rotate_log_event`,
  `Table_map_log_event`, and `Rows_log_event`.
- `vendor/mariadb/server/sql/sql_binlog.cc` implements the internal `BINLOG`
  statement by decoding base64 binlog events and applying them through replay
  context. MariaDB documents `BINLOG` as an internal-use statement generated by
  `mariadb-binlog` for re-applying events.
- `vendor/mariadb/server/sql/sql_parse.cc` routes server-oriented binlog and
  replication surfaces, including `COM_BINLOG_DUMP`, `SHOW BINLOG EVENTS`,
  `SHOW BINLOG STATUS`, `SHOW BINARY LOGS`, `PURGE BINARY LOGS`, and
  `SQLCOM_BINLOG_BASE64_EVENT`.
- MariaDB documents the binary log as a set of server log files for data
  changes, replication, and point-in-time recovery. Those sidecar files do not
  fit MyLite's default single-primary-file product shape.
- MariaDB documents `BINLOG` as an internal-use statement generated by the
  `mariadb-binlog` utility to replay binary-log events:
  https://mariadb.com/docs/server/reference/sql-statements/administrative-sql-statements/binlog
- MariaDB documents binary-log inspection and maintenance through
  `SHOW BINLOG EVENTS` and related server administration statements:
  https://mariadb.com/docs/server/reference/sql-statements/administrative-sql-statements/show/show-binlog-events

the real binlog code from a linked artifact.

```text
libmariadbd.a: 32,107,110 bytes
```

## Proposed Design

Do not remove `binlog_tp`, `mysql_bin_log`, `log.cc`, or the `binlog` daemon
plugin in this slice. That would require a wider transaction-manager refactor,
including XA, savepoint, row-logging, GTID, and startup interactions. A
half-removal would be easy to build incorrectly because many runtime checks are
direct `&binlog_tp` comparisons rather than plugin lookups.

Use this slice to establish the safe boundary:

1. Keep binary logging disabled in the embedded runtime defaults and continue
   verifying that no binlog, relay-log, or GTID-index sidecar appears in MyLite
   runtime directories.
2. Treat user-facing binlog and replication SQL surfaces as unsupported in the
   minsize embedded profile. Where a small source-local stub can replace a
   server-only implementation without touching transaction internals, prefer a
   stable unsupported diagnostic.
3. Re-test link-time section garbage collection after the earlier
   executable-export profile. The previous section-GC experiment predated the
   removal of `ENABLE_EXPORTS` from MyLite smoke executables; now that the
   linked binary no longer exports nearly every symbol, section GC may be able
   to discard unreferenced binlog/replay functions inside otherwise required
   objects.

The first implementation attempt after this spec should therefore be a
bounded build-profile experiment, not direct deletion of `log.cc` internals:

```text
MYLITE_ENABLE_SECTION_GC=ON
```

When enabled for minsize builds, compile embedded-library targets with
function/data sections and link runtime artifacts with `--gc-sections` in
addition to the existing lld RELR flags. Keep the option easy to revert if the
archive growth outweighs linked-runtime savings.

## Affected MariaDB Subsystems

- Embedded build profile and linker flags.
- SQL/logging code only indirectly through dead-code elimination.
- Binlog and replication command surfaces remain unsupported for MyLite's
  default embedded product, but this slice does not attempt a full parser or
  command-table removal.
- Transaction manager behavior must remain unchanged: MyLite DML commit,
  rollback, statement rollback, and savepoint tests are the main regression
  guard.

## DDL Metadata Routing Impact

No MyLite table-definition metadata change. Binlog and replication commands
must not create MyLite catalog objects or durable MariaDB datadir sidecars.

## Single-File and Embedded-Lifecycle Impact

The default runtime must continue to avoid persistent binlog, relay-log,
GTID-index, and replication metadata sidecars. Existing MyLite sidecar scans
should remain the guard.

Opening and closing a `.mylite` file must continue to work without enabling
file-based binary logging.

## Public API and File-Format Impact

No public `libmylite` C API change and no MyLite file-format change.

The SQL compatibility surface remains intentionally smaller than MariaDB Server
for server replication/binlog administration.

## Binary-Size Impact

The maximum direct archive bytes in the currently identified binlog/replay
object cluster are roughly 512 KiB before compression, excluding the shared
`rpl_utility_server.cc.o` and broader SQL/type objects. Full removal is not a
near-term safe slice because `log.cc` owns both general logging infrastructure
and the binlog transaction participant.

The section-GC attempt may reduce the linked runtime more than the archive.
Archive size may grow because `-ffunction-sections` and `-fdata-sections`
increase object metadata. The attempt is worth re-measuring only because the
exported-symbol profile changed since the last rejected section-GC experiment.

Measured implementation result:

```text
libmariadbd.a: 32,107,110 -> 36,520,566 bytes (+4,413,456)
stripped mylite-open-close-smoke: 8,786,856 -> 8,458,680 bytes (-328,176)
```

The attempt is a linked-runtime-size win and a static-archive-size loss. The
full compatibility harness passed with no unexpected sidecars.

## License, Trademark, and Dependency Impact

No new dependencies. All affected code remains MariaDB-derived GPL-2.0-only
source already present in the vendor tree.

## Acceptance Criteria

- The slice does not destabilize MyLite transaction commit, rollback,
  statement rollback, savepoint, storage, or compatibility harness behavior.
- Any accepted size flag reduces the stripped linked runtime materially or is
  documented as a rejected attempt.
- Production size analysis records the measured archive and linked-runtime
  deltas.
- The repository does not claim full binlog removal while `log.cc`, `binlog_tp`,
  and `mysql_bin_log` remain linked.

## Risks and Unresolved Questions

- `log.cc` mixes general logging, transaction coordination, binlog state, GTID,
  and event-writing code. Meaningful source-level removal needs a larger
  refactor that splits no-binlog transaction coordination from file-based
  binary logging.
- MariaDB defines `HAVE_REPLICATION` unconditionally in
  `vendor/mariadb/server/include/my_global.h`, so treating replication as a
  build-time feature is not currently a simple cache option.
- Section GC may still fail to save meaningful bytes if references from
  `log.cc`, `handler.cc`, `sql_parse.cc`, and `sys_vars.cc` keep most sections
  live.
- Some server-only SQL surfaces are parsed even when the runtime cannot execute
  them usefully. Removing parser grammar for those surfaces is a separate
  compatibility and grammar-size slice.
