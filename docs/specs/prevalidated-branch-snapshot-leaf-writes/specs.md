# Prevalidated Branch Snapshot Leaf Writes

## Problem

Prepared insert profiling after active branch-page caching shows the largest
remaining branch execution cost in `refold_branch_index_root_insert()` and
`write_index_branch_snapshot_pages()`. Branch snapshot writes generate leaf
pages through `prepare_index_branch_snapshot_pages()`, then the generic pager
write path decodes and checksums those same freshly encoded leaf pages only to
refresh the active leaf-page cache.

## Source Findings

- Base line: MariaDB 11.8.6, commit
  `9bfea48ce1214cc4470f6f6f8a4e30352cef84e7`.
- MariaDB refs:
  - `mariadb/sql/sql_insert.cc:Write_record::single_insert()` executes prepared
    row inserts through `handler::ha_write_row()`.
  - `mariadb/sql/handler.cc:handler::ha_write_row()` delegates accepted writes
    to the storage engine.
  - `mariadb/storage/mylite/ha_mylite.cc:ha_mylite::write_row()` routes MyLite
    row writes to `mylite_storage_append_row_with_index_entries()`.
- MyLite refs:
  - `packages/mylite-storage/src/storage.c:prepare_index_branch_snapshot_pages()`
    builds a branch snapshot from freshly encoded leaf pages and a branch page.
  - `packages/mylite-storage/src/storage.c:write_index_branch_snapshot_pages()`
    writes those generated leaf pages followed by the branch page.
  - `packages/mylite-storage/src/storage.c:store_active_index_leaf_page_from_pager_write()`
    decodes and checksums arbitrary leaf-page pager writes before updating the
    active leaf-page cache.

## Design

- Add a narrow pager write helper for prevalidated index leaf pages generated by
  branch snapshot preparation.
- Keep generic `pager_write_page()` behavior unchanged for arbitrary caller
  writes.
- For branch snapshot leaf pages, write the page, then refresh the active
  leaf-page cache from encoded page metadata already present in the generated
  bytes. Check the page type, stored page id, key size, entry count, and used
  byte bounds before caching, but do not rerun checksum validation over the
  freshly encoded page.
- Keep branch page writes on the generic pager path because their active branch
  cache refresh still needs generic validation.

## Compatibility Impact

No SQL, public API, storage-routing, metadata, or file-format behavior changes.
The slice only removes redundant cache-refresh validation after first-party
branch snapshot encoding.

## DDL Metadata Routing Impact

None. Catalog publication and table-definition routing are unchanged.

## Single-File And Embedded Lifecycle

The helper writes the same pages to the same primary `.mylite` file and does
not add companion files. It preserves dirty-page undo capture and active dirty
buffer discard behavior from the generic pager write path.

## Public API Or File-Format Impact

None.

## Storage-Engine Routing Impact

No routing-policy change. Routed `InnoDB`, `MyISAM`, `Aria`, explicit MyLite,
and supported aliases continue to use the same MyLite storage path.

## Wire-Protocol Or Integration Impact

None.

## Binary-Size, License, And Dependency Impact

No new dependency or license impact. Binary-size impact is limited to one small
helper and test hook coverage.

## Acceptance Criteria

- Branch snapshot leaf writes refresh active leaf-page cache entries without
  calling the generic decode/checksum cache-refresh path.
- Generic pager writes keep their existing decode/checksum validation.
- Dirty-page undo capture and active dirty-buffer discard behavior are
  preserved.
- Existing maintained-index correctness and rollback coverage still passes.
- Prepared insert component profiling moves cost away from active leaf-cache
  decode on branch snapshot writes or records the next bottleneck.

## Risks

- Bypassing decode for arbitrary pages would hide corruption. The helper is
  therefore only used for branch snapshot pages generated immediately before
  writing and still checks page identity and bounds before caching.
