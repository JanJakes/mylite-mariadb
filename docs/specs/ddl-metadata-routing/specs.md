# ddl-metadata-routing

## Problem Statement

MyLite can discover a seed table definition from engine-owned metadata, but
user DDL still has no MyLite-owned destination for table definitions. The next
slice should prove that MariaDB can run a bounded `MYLITE` table DDL lifecycle
without leaving durable `.frm` table-definition sidecars.

Because the durable single-file catalog does not exist yet, this slice should
store DDL-created table definitions in a process-local engine catalog as copied
binary frm images. That keeps the MariaDB DDL integration testable while
avoiding a false claim that `.mylite` file-format persistence exists.

## Scope

- Capture `CREATE TABLE ... ENGINE=MYLITE` table definitions in the `MYLITE`
  handler from the frm image already generated by MariaDB's SQL layer.
- Discover DDL-created tables from copied frm images with
  `TABLE_SHARE::init_from_binary_frm_image(..., write=false)`.
- Support the bounded smoke lifecycle:
  - `CREATE TABLE mylite.created (id INT) ENGINE=MYLITE`
  - `ALTER TABLE mylite.created ADD COLUMN note VARCHAR(12), ALGORITHM=COPY`
  - `RENAME TABLE mylite.created TO mylite.renamed`
  - `DROP TABLE mylite.renamed`
- Preserve the existing seed table `mylite.probe`.
- Fail the storage-engine smoke if any `.frm` file remains in the runtime.
- Record DDL lifecycle observations in the smoke report.

## Non-Goals

- Do not implement the durable MyLite catalog.
- Do not persist schema definitions in the `.mylite` primary file.
- Do not implement row storage or data-copy semantics beyond empty scans.
- Do not support in-place or online ALTER as a MyLite storage feature.
- Do not remove MariaDB's temporary compatibility datadir or schema directory
  requirements in this slice.
- Do not implement crash recovery for partially completed DDL.
- Do not add public `libmylite` API.

## Source Findings

- Base source: MariaDB Server `mariadb-11.8.6`
  (`9bfea48ce1214cc4470f6f6f8a4e30352cef84e7`).
- `sql/sql_table.cc:mysql_create_frm_image()` builds the binary frm image for
  SQL-layer `CREATE TABLE` metadata before the handler create call.
- `sql/sql_table.cc:create_table_impl()` passes that image to
  `ha_create_table()` for ordinary create and copy ALTER table creation.
- `sql/handler.cc:ha_create_table()` sets `write_frm_now` to false when the
  target engine has `handlerton::discover_table`, then initializes a temporary
  `TABLE_SHARE` from the frm image without writing the `.frm` file.
- `sql/table.cc:TABLE_SHARE::init_from_binary_frm_image()` writes a `.frm` only
  when its `write` argument is true. Passing false parses the image in memory.
- `sql/table.h` documents `TABLE_SHARE::frm_image` as available during
  `CREATE TABLE`; this gives the handler a narrow source for the definition
  image.
- `sql/handler.cc:ha_delete_table()` calls `handlerton::drop_table()`. The
  default handlerton drop path instantiates a handler and calls
  `handler::delete_table()`.
- `sql/sql_table.cc:mysql_rm_table_no_locks()` treats discovery engines
  specially when deleting a `.frm`: if a `.frm` exists for a discovery engine,
  it is deleted silently; if none exists, drop can still proceed through the
  engine.
- `sql/sql_table.cc:mysql_rename_table()` calls `handler::ha_rename_table()`
  when `QRMT_HANDLER` is set. If the engine rename succeeds but `.frm` rename
  returns `ENOENT`, MariaDB treats that as acceptable for handlers with no
  `.frm` sidecar.
- `sql/sql_table.cc:create_table_for_inplace_alter()` still calls
  `init_from_binary_frm_image(..., true, ...)` for in-place ALTER candidate
  setup. This writes a temporary frm path before the handler can reject
  in-place ALTER. The bounded smoke should use `ALGORITHM=COPY` while the
  final catalog design decides how to remove or intercept this transient path.
- `sql/sql_table.cc:alter_copy` creates the intermediate table through
  `ha_create_table()`, so discovery-enabled engines can avoid writing the
  intermediate `.frm` in the copy ALTER path.

## Proposed Design

Replace the single hard-coded seed definition with a small process-local
catalog inside `ha_mylite.cc`. Each catalog entry should contain:

- database name,
- table name,
- optional SQL seed string,
- optional copied binary frm image.

The initial catalog contains `mylite.probe` from the existing seed SQL string.
DDL-created entries use copied frm bytes.

`ha_mylite::create()` should validate that `TABLE_SHARE::frm_image` is present
and copy it into the catalog under `TABLE_SHARE::db` and
`TABLE_SHARE::table_name`. It should return a table-exists error if an entry
already exists.

`ha_mylite::delete_table()` should parse the database and table name from the
handler path and remove the matching catalog entry. Missing entries should be
reported as table-not-found so MariaDB can apply its existing DROP handling.

`ha_mylite::rename_table()` should parse source and destination database/table
names from handler paths and move the catalog entry without changing the copied
frm image.

Discovery should look up the requested table in the catalog:

- for frm-backed entries, initialize the `TABLE_SHARE` from the stored binary
  image with `write=false`;
- for seed SQL entries, keep using
  `TABLE_SHARE::init_from_sql_statement_string(..., write=false)`.

The smoke should verify that `CREATE`, copy `ALTER`, `RENAME`, and `DROP`
succeed and that the final table list returns to the seed table only.

## Affected Subsystems

- MyLite storage-engine handler and handlerton discovery hooks.
- Storage-engine smoke executable and wrapper script.
- Roadmap and slice documentation.

No public API, row-storage, file-format, or embedded-open behavior changes.

## DDL Metadata Routing Impact

This slice creates the first MyLite-owned destination for MariaDB DDL table
definitions, but it is deliberately process-local. It proves that the SQL layer
and handler boundary can avoid durable `.frm` table-definition files for the
bounded lifecycle.

It does not solve final DDL persistence. The next catalog slice must move the
same logical definition records into the primary `.mylite` file, add versioning
and recovery rules, and replace the temporary schema-directory compatibility
artifact.

## Single-File And Embedded-Lifecycle Implications

The process-local catalog is lost when the embedded runtime exits. It is useful
only as a DDL routing proof. The smoke still creates a temporary schema
directory under the smoke datadir because MariaDB's database namespace still
expects it. That directory is not a final MyLite sidecar contract.

## Public API Or File-Format Impact

None. No durable `.mylite` catalog format is introduced.

## Binary-Size Impact

The slice adds small first-party storage-engine code and smoke assertions.
Record artifact sizes after implementation.

## Test And Verification Plan

- Run `tools/run-storage-engine-smoke.sh`.
- Verify the report records successful DDL create, alter, rename, and drop
  phases.
- Verify `mylite.created` can be scanned after `CREATE`.
- Verify `SHOW COLUMNS FROM mylite.created LIKE 'note'` succeeds after copy
  `ALTER`.
- Verify `mylite.renamed` can be scanned after `RENAME`.
- Verify `mylite.renamed` disappears after `DROP`.
- Verify no `.frm` files remain in the storage-engine smoke runtime.
- Verify dynamic plugin artifacts remain absent.
- Run `tools/run-libmylite-open-close-smoke.sh`.
- Run `tools/run-embedded-bootstrap-smoke.sh`.
- Run `bash -n` for changed shell scripts.
- Run `git diff --check`.

## Acceptance Criteria

- `CREATE TABLE ... ENGINE=MYLITE` succeeds for the smoke table and stores the
  table definition in the MyLite engine catalog.
- Copy `ALTER TABLE ... ALGORITHM=COPY` succeeds for the smoke table and keeps
  the altered definition discoverable without a durable `.frm`.
- `RENAME TABLE` and `DROP TABLE` update the MyLite engine catalog.
- The storage-engine smoke ends with only the seed `mylite.probe` table
  discoverable.
- No `.frm` file remains in the smoke runtime.
- Existing embedded and `libmylite` smokes still pass.

## Risks And Unresolved Questions

- The process-local catalog is not durable and is not a file-format design.
- Default in-place ALTER candidate setup can still create a transient `.frm`
  before the handler's final algorithm decision. This slice avoids that path
  with explicit `ALGORITHM=COPY`; a later slice must either intercept that path
  or make MyLite force copy routing before temporary frm creation.
- Path parsing is intentionally limited to MariaDB's simple smoke paths. The
  durable catalog slice should stop deriving metadata identity from filesystem
  paths.
- Schema-directory elimination needs a broader namespace and catalog design.
