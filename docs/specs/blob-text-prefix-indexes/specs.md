# BLOB/TEXT Prefix Indexes

## Problem

MyLite can store BLOB/TEXT row payloads and can persist ordinary primary,
unique, and secondary index-entry pages, but it still rejects any key part with
MariaDB's `HA_BLOB_PART` flag. That blocks valid MySQL/MariaDB schemas that use
bounded prefix indexes on `TEXT`, `LONGTEXT`, `BLOB`, or `LONGBLOB` columns.

The rejection is now broader than the actual storage limitation. MyLite already
stores MariaDB key-tuple bytes generated by MariaDB itself, and MariaDB's key
helpers know how to create and compare BLOB/TEXT prefix key images.

## Source Findings

MariaDB base: `mariadb-11.8.6`
(`9bfea48ce1214cc4470f6f6f8a4e30352cef84e7`).

- `mariadb/sql/table.cc` builds `KEY_PART_INFO` from fields. BLOB/TEXT fields
  contribute `HA_BLOB_PART`, add `HA_KEY_BLOB_LENGTH` bytes of key-part length
  metadata, and mark partial key segments when the declared key length is a
  prefix.
- `mariadb/sql/field.h:Field_blob` reports BLOB/TEXT key parts as
  `HA_KEYTYPE_VARTEXT2` or `HA_KEYTYPE_VARBINARY2`, returns `HA_BLOB_PART`, and
  adds `HA_KEY_BLOB_LENGTH` bytes to the key image.
- `mariadb/sql/field.cc:Field_blob::get_key_image_itRAW()` stores a two-byte
  key-image length followed by a character-boundary-safe prefix of the
  BLOB/TEXT value bytes.
- `mariadb/sql/field.cc:Field_blob::key_cmp()` compares stored BLOB/TEXT key
  images with the field collation and the requested key length.
- `mariadb/sql/key.cc:key_copy()` copies BLOB/TEXT key parts by asking the
  field for its key image, so the handler does not need to inspect BLOB
  pointers directly for index entries.
- `mariadb/sql/key.cc:key_tuple_cmp()` and `mariadb/sql/key.cc:key_buf_cmp()`
  compare variable text/binary key images, including BLOB/TEXT key types and
  null-byte prefixes.
- `mariadb/sql/sql_class.cc` rejects BLOB/TEXT key parts unless the handler
  advertises `HA_CAN_INDEX_BLOBS`, so MyLite must expose that capability before
  `CREATE TABLE` or copy-rebuild index DDL can reach `ha_mylite::create()`.
- MariaDB documentation for `CREATE TABLE` and `CREATE INDEX` describes index
  columns as allowing an optional prefix length:
  <https://mariadb.com/docs/server/server-usage/tables/create-table> and
  <https://mariadb.com/kb/en/create-index/>.
- MariaDB's BLOB/TEXT documentation describes BLOB/TEXT indexability:
  <https://mariadb.com/docs/server/reference/data-types/string-data-types/blob-and-text-data-types>.

## Scope

- Bounded BLOB/TEXT prefix indexes over MyLite-routed base tables.
- Plain and unique BTREE-compatible prefix indexes declared in `CREATE TABLE`.
- Standalone copy-rebuild `CREATE INDEX ... ALGORITHM=COPY` and
  `CREATE UNIQUE INDEX ... ALGORITHM=COPY` over supported BLOB/TEXT prefix key
  parts.
- Insert, duplicate-key checks, forced-index lookup, update/delete visibility,
  close/reopen visibility, and durable-sidecar gates.
- Routed requests recorded as omitted/default, `MYLITE`, `InnoDB`, `MyISAM`,
  or `Aria` when the rest of the table shape is supported.

## Non-Goals

- Full-length unbounded BLOB/TEXT indexes that exceed the current single
  index-entry page capacity.
- FULLTEXT, SPATIAL, vector, generated/expression, hash, or foreign-key
  indexes.
- Key-only reads, index condition pushdown, multi-range read optimization, or
  persistent B-tree pages.
- Physical compaction of stale row, overflow, or index-entry pages.
- SQL rollback, savepoints, or transaction-aware index maintenance.

## Design

Keep key extraction and comparison at the MariaDB handler boundary:

1. Continue using `key_copy()` to generate key-tuple bytes from the MariaDB row
   buffer. For BLOB/TEXT prefix parts this stores MariaDB's normal key image,
   not the row-buffer process pointer.
2. Keep storing opaque key-tuple bytes in MyLite index-entry pages.
3. Widen the MyLite index support predicate to accept `HA_BLOB_PART` key parts
   when the key remains an ordinary BTREE-compatible key and the total
   `KEY::key_length` fits MyLite's current index-entry page.
4. Continue using `key_tuple_cmp()` for cursor ordering and range positioning.
5. Continue using `key_buf_cmp()` for unique-key duplicate checks, including
   BLOB/TEXT prefix uniqueness.

This is not a new index structure. It is a support-gate correction plus tests
for a key class MariaDB already represents as bounded key-tuple bytes.

## Compatibility Impact

BLOB/TEXT prefix indexes move from planned to partial for supported routed
table shapes. The behavior remains partial because FULLTEXT/SPATIAL indexes,
unbounded or oversized key images, physical B-tree navigation, SQL rollback,
foreign-key interactions, and transaction-aware index maintenance remain
unsupported.

Unique BLOB/TEXT prefix indexes enforce uniqueness of the indexed prefix, which
matches MariaDB's key-image semantics for prefix key parts.

## DDL Metadata Routing Impact

`CREATE TABLE`, copy `ALTER`, and standalone copy-rebuild index DDL can publish
catalog metadata for supported BLOB/TEXT prefix indexes. Unsupported key
classes must still fail before catalog publication. Requested engine metadata
continues to be preserved separately from effective `MYLITE` routing.

## Single-File And Embedded Lifecycle Impact

All key bytes continue to live in primary-file index-entry pages. No `.frm`,
`.ibd`, `.MYD`, `.MYI`, `.MAI`, `.MAD`, Aria log, binlog, relay log, or
plugin-owned durable table file is introduced. Existing rollback-journal and
sidecar cleanup behavior remains in force.

## Public API And File Format Impact

No public `libmylite` API changes. No new storage page type is required. The
current index-entry page already stores variable-size key bytes; this slice
only allows additional MariaDB key images within the existing page capacity.

## Storage-Engine Routing Impact

The support applies through the existing MyLite handler to routed table
definitions regardless of whether SQL requested `MYLITE`, omitted/default
engine, `InnoDB`, `MyISAM`, or `Aria`. External durable engine files remain
out of scope.

## Binary-Size And Dependency Impact

No dependency is added. Binary-size impact should be limited to support-gate
and test code; update size measurements after verification.

## Acceptance Criteria

- Supported BLOB/TEXT prefix-index DDL succeeds before catalog publication.
- Forced-index reads find rows through persisted BLOB/TEXT prefix key entries
  before and after close/reopen.
- Unique prefix indexes reject duplicate indexed prefixes and allow distinct
  prefixes.
- Update, delete, and truncate visibility continues to filter stale prefix
  index entries through row-state pages.
- Compatibility, roadmap, storage architecture, and application-schema docs
  describe BLOB/TEXT prefix indexes as partial support with explicit remaining
  limits.

## Implementation Status

Implemented in the MyLite handler:

- `ha_mylite::table_flags()` advertises `HA_CAN_INDEX_BLOBS`, allowing
  MariaDB's DDL validator to accept BLOB/TEXT prefix key parts for MyLite.
- `mylite_key_is_supported()` accepts bounded BLOB/TEXT key parts when the
  whole key image fits the current MyLite index-entry page, while continuing to
  reject unsupported algorithms and unbounded zero-length BLOB/TEXT parts.
- Storage-engine smoke coverage checks declared and standalone BLOB/TEXT
  prefix indexes, unique prefix duplicate enforcement, update/delete filtering,
  close/reopen persistence, and durable-sidecar gates.
- Application-schema smoke now includes `meta_key(191)` on a wider
  `varchar(255)` and a bounded `text` prefix index.

## Risks And Open Questions

- Prefix key semantics are byte and collation sensitive. The handler should
  keep delegating key images and comparison to MariaDB rather than copying SQL
  type logic into MyLite storage.
- The current append-only cursor path is O(n) over live index entries. This is
  acceptable for compatibility coverage but not the final performance design.
- Very large prefix lengths can exceed one MyLite index-entry page. Those must
  remain unsupported until the storage format has multi-page index key entries
  or physical index pages.
