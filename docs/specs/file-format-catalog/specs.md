# File Format And Catalog

## Problem Statement

MyLite currently creates a primary `.mylite` path, starts a temporary MariaDB
runtime, and proves SQL execution plus storage-engine registration. It does not
store durable catalog or table metadata in the primary file.

This slice defines the first durable file-format and catalog contract. It is the
prerequisite for metadata discovery and DDL routing: MariaDB must be able to
open a MyLite table from catalog metadata without relying on persistent `.frm`
files or any existing engine sidecars.

## Scope

- Define the initial `.mylite` file header and catalog page shape.
- Define catalog records for schemas, tables, MariaDB table definitions, and
  storage roots.
- Define how the MyLite handler stores MariaDB-generated table-definition
  images during DDL.
- Define how the MyLite handler discovers table names, table existence, and
  `TABLE_SHARE` definitions from the catalog.
- Define the verification plan for durable catalog create, close, reopen, and
  sidecar detection.

## Non-Goals

- Do not implement row pages, indexes, transactions, recovery, locking, or
  multi-writer behavior in this slice.
- Do not route `ENGINE=InnoDB`, `ENGINE=MyISAM`, or `ENGINE=Aria` to MyLite.
- Do not implement `ALTER TABLE`, `DROP TABLE`, `RENAME TABLE`, views, triggers,
  routines, foreign keys, or generated-column storage.
- Do not claim binary compatibility for arbitrary external `.frm` files.
- Do not expose the catalog format as a public API.

## Proposed Design

### Primary File Header

Page 0 is the fixed header page. It contains:

- magic bytes and header version;
- byte-order marker;
- page size;
- file-format major and minor versions;
- MyLite library compatibility version;
- feature flags;
- active catalog root page id;
- current transaction or catalog generation id;
- free-list or allocator root page id;
- checksum algorithm id and header checksum.

The first implementation should create a valid header and empty catalog on
`mylite_open(..., MYLITE_OPEN_CREATE)` instead of creating an empty placeholder
file. The exact binary constants belong in the implementation header, but the
header must be fixed-size, endian-marked, checksummed, and forward-versioned
from the first durable write.

Implementation status: MyLite now writes a fixed 4096-byte header page and a
catalog root page, validates magic bytes, format version, endian marker,
checksums, page count, and catalog metadata, and maps unsupported storage
formats to a corrupt open result through `libmylite`. The storage package also
stores explicit MyLite table-definition records and checksummed definition blob
pages.

### Catalog Records

The catalog is the authoritative metadata store. The first record families are:

- schemas: catalog namespace id, schema name, default character set and
  collation;
- tables: table id, schema id, table name, requested engine, effective MyLite
  engine, table-definition version, flags, and root page ids;
- table-definition blobs: the MariaDB-generated binary table-definition image
  plus optional partition image;
- table SQL text: optional original or canonical SQL text for diagnostics and
  future rebuild tooling;
- storage roots: placeholder row root, index root, autoincrement state, and
  overflow-blob root ids.

For the first metadata slice, table-definition blobs may be the only complete
source for columns and indexes. The catalog still reserves normalized records for
columns, indexes, constraints, and foreign keys so later row/index work can avoid
changing the top-level catalog model.

### DDL Capture

For `CREATE TABLE ... ENGINE=MYLITE`, the MyLite handler should let MariaDB
produce the canonical table-definition image, then store that image in the
catalog before reporting success.

The expected create path is:

1. MariaDB resolves the storage engine and builds the table-definition image via
   `mysql_create_frm_image()`.
2. Because MyLite implements `discover_table`, `ha_create_table()` must not
   persist a `.frm` file as application metadata.
3. `ha_mylite::create()` reads the generated image from `TABLE_SHARE` with
   `read_frm_image()`.
4. MyLite writes a catalog transaction containing schema, table, and
   table-definition records.
5. The handler rejects table storage operations until row storage exists.

This slice should support only an empty durable table definition if row storage
is not ready. Row-modifying DML against such a table must fail explicitly rather
than silently creating rows elsewhere.

### Discovery

The MyLite handlerton should implement:

- `discover_table_existence`: check the catalog namespace and table index
  without loading the full definition;
- `discover_table_names`: enumerate catalog tables for `SHOW TABLES` and
  information schema;
- `discover_table`: load the stored binary table-definition image from the
  catalog and call `TABLE_SHARE::init_from_binary_frm_image(thd, false, ...)`.

The `write` argument must be false for normal discovery so a persistent `.frm`
cache is not created. A future performance slice may add a MyLite-owned
definition cache only if its lifecycle is documented and sidecar tests prove it
is not user-managed durable state.

### Catalog Transactions

The first implementation can use a single-writer catalog update path, but the
file format must not bake in a single-writer-only design. Catalog updates should
be represented as publishable generations: new or updated metadata pages are
written first, checksummed, and then made visible by an atomic root/generation
update. Crash recovery details belong to the transactions and recovery slice,
but the header and catalog page format must leave room for it now.

### Compatibility Policy

`ENGINE=MYLITE` becomes a table-definition compatibility surface only after the
metadata slice lands. It still does not imply row DML, indexes, transactions, or
engine routing from other engines. Unsupported statements must fail with stable
MyLite result codes and MariaDB diagnostics instead of falling back to MyISAM,
Aria, InnoDB, or ordinary `.frm` files.

## Affected MariaDB Subsystems

- Storage-engine handlerton registration in `mariadb/storage/mylite/`.
- Handler `create()`, `delete_table()`, and later `rename_table()`/partition
  metadata hooks.
- MariaDB table discovery through `discover_table`,
  `discover_table_names`, and `discover_table_existence`.
- DDL creation in `sql_table.cc` and `handler.cc`.
- Table definition cache behavior through `TABLE_SHARE`.

## DDL Metadata Routing Impact

This is the first slice that changes MyLite from "engine can be discovered" to
"metadata can be durable." It must route `CREATE TABLE ... ENGINE=MYLITE`
metadata into the MyLite catalog and must prove no persistent `.frm` is left as
application metadata.

`DROP`, `RENAME`, and `ALTER` should remain unsupported until their catalog
transaction semantics are designed. If MariaDB calls those paths for cleanup
after a failed create, the handler must remove any incomplete catalog records.

## Single-File And Embedded Lifecycle

Durable metadata lives in the primary `.mylite` file. The temporary MariaDB
runtime directory remains bootstrap debt and must still be removed on final
close. Tests must detect persistent `.frm`, `.MYD`, `.MYI`, `.MAI`, `.MAD`,
`.ibd`, redo, undo, and Aria control sidecars near the primary file and runtime
root.

The catalog reader must work after `mylite_close()` followed by a fresh
`mylite_open()`. It must not depend on process-local MariaDB state.

## Public API And File-Format Impact

The public C API does not need new calls for this slice. File-format versioning
does become visible operationally:

- older libraries must reject newer incompatible files with `MYLITE_CORRUPT` or
  a more specific future result code;
- newer libraries must either open compatible older formats or fail explicitly;
- diagnostics should identify unsupported file-format versions.

The format constants should live in first-party MyLite code, not in upstream
MariaDB files.

## Storage-Engine Routing Impact

The only engine request affected by this slice is explicit `ENGINE=MYLITE`.
Routing `ENGINE=InnoDB`, `ENGINE=MyISAM`, `ENGINE=Aria`, and no-engine DDL to
MyLite remains a later compatibility slice because those paths require policy
for engine-specific options and unsupported semantics.

Implementation status: explicit `CREATE TABLE ... ENGINE=MYLITE` now stores the
MariaDB-generated table-definition image in the primary `.mylite` file and uses
catalog-backed `discover_table`, `discover_table_names`, and
`discover_table_existence`. Row-modifying DML, `DROP`, and `RENAME` still fail
through unsupported handler paths until their catalog transaction semantics are
specified.

## Wire Protocol Or Integration Impact

None directly. Future wire-protocol packages should observe the same SQL and
diagnostic behavior through `libmylite`; they should not bypass catalog routing.

## Binary Size Impact

The catalog code lives in first-party MyLite packages and the MyLite handler.
The expected size increase is small compared with the embedded MariaDB archive,
but the implementation slice must still record archive size if it changes the
MariaDB embedded build or enables new MariaDB components.

## License Or Dependency Impact

No new dependency is expected. MyLite first-party catalog code remains in the
GPL-2.0 project. Stored MariaDB table-definition images are runtime metadata
generated by MariaDB and are not source imports.

## Test And Verification Plan

1. Unit-test header initialization, magic/version validation, checksum failure,
   and unsupported-version rejection.
2. Unit-test empty catalog creation, schema/table lookup, duplicate-table
   rejection, and missing-table lookup.
3. Add embedded SQL coverage for `CREATE TABLE t (...) ENGINE=MYLITE` once
   `ha_mylite::create()` stores catalog metadata.
4. Close and reopen the database, then verify `SHOW TABLES` and table discovery
   use catalog metadata.
5. Verify no persistent `.frm` or existing-engine sidecars are created for
   MyLite tables.
6. Verify row-modifying DML against a metadata-only MyLite table fails
   explicitly until row storage lands.
7. Verify failed create cleanup does not leave a half-published catalog record.
8. Run dev, embedded, and storage-smoke CMake presets.

## Acceptance Criteria

- A newly created `.mylite` file has a validated header and empty catalog.
- The MyLite catalog stores schema and table metadata for explicit
  `ENGINE=MYLITE` DDL.
- The catalog stores MariaDB-generated table-definition images and uses them for
  `discover_table`.
- `SHOW TABLES` and table-existence checks use MyLite catalog enumeration.
- Close/reopen preserves catalog metadata without `.frm` files.
- Existing-engine durable sidecars are not created.
- Unsupported row-modifying DML and unsupported DDL fail explicitly.
- `docs/COMPATIBILITY.md` marks `ENGINE=MYLITE` metadata as partial only after
  tests land.

## Risks And Unresolved Questions

- Binary table-definition images are a pragmatic first bridge, not a complete
  semantic catalog. Later slices must decide which column, index, and constraint
  fields are materialized for storage execution.
- MariaDB `.frm` image compatibility is tied to the imported MariaDB baseline.
  File-format upgrades must account for future MariaDB imports.
- The first catalog transaction mechanism may be coarse, but it must not block
  the long-term write-concurrency goal.
- Partition metadata needs explicit treatment before partitioned tables can be
  supported.
- The policy for default engine routing remains unresolved and should not be
  folded into this slice.
